\section{Empirical Evaluation}
\label{sec:eval}

We evaluate the AST surface along six axes: (i) cross-language
compression ratios on a realistic benchmark fixture; (ii) outline
token-budget conformance on a wider corpus; (iii) outline emission
latency; (iv) outline-discipline adoption trajectory; (v) symbolic
surgery false-negative rate; (vi) end-to-end token reduction on real
agent traces.

\paragraph{What is measured now vs.\ pending.}
Three of the six claims are currently measured:
\begin{itemize}[leftmargin=*,nosep]
  \item \textbf{Compression ratios} (\cref{sec:eval:compression}) are
        measured on a 40-file polyglot corpus (10 files per language)
        drawn from open-source repositories (\cref{sec:eval:setup}) and
        backed by a runnable CSV pipeline.
  \item \textbf{Token-budget conformance} (\cref{sec:eval:budget}) and
        \textbf{outline emission latency} (\cref{sec:eval:latency}) are
        measured on Phase~1a (Go-only, three repositories, 600 files
        sampled), with Python/JS/TS deferred to Phase~1b.
\end{itemize}
The remaining three axes correspond to pre-registered open questions
OQ-AST-3 through OQ-AST-5 (\cref{app:open-questions}) whose datasets
are still accumulating:
OQ-AST-3 (adoption trajectory) needs at least 14~days of federated
outline-events telemetry;
OQ-AST-4 (compile-gate false-negative rate) needs a curated 200-diff
refactor corpus;
OQ-AST-5 (end-to-end token reduction) needs paired Explore-tier
traces with and without outline discipline enabled.
These are deferred to follow-up work and are not considered failures
of the current system --- they are the pre-registered programme that
will promote, scope, or demote subsystems as data arrive.

This section is structured as a \emph{reproducibility-first} report:
every quantitative claim is backed by a CSV in \texttt{data/} and a
script in \texttt{scripts/} (\cref{sec:repro}).

\subsection{Setup and corpus}
\label{sec:eval:setup}

\paragraph{Bench corpus (v2).}
The compression-ratio measurement uses a 40-file polyglot corpus
(10 files per language) in \texttt{scripts/corpus-files/\{go,py,js,tsx\}/}.
Files were drawn from four open-source repositories already cloned in
\texttt{data/scratch/corpora/} and from the Blackrim checkout, chosen
to represent varied shapes: API handler, data model, test harness, CLI
command, utility, and domain-specific computation.
LoC ranges: Go 76--1098 (median 225), Python 68--370 (median 118),
JavaScript 60--125 (median 81), TypeScript 58--248 (median 83).
The measurement uses a structural-outline approximation
(declaration-header lines retained, body lines dropped) implemented in
\texttt{pull-compression-ratios.py}; this is consistent with the
\texttt{gt outline} AST walk for the declaration surface the paper
focuses on.\footnote{Prior to v2 ($n{=}1$ per language, single in-tree
bench fixture), measured token savings were: Go 82.0\%, Python 84.1\%,
JavaScript 83.0\%, TypeScript 75.2\%
(\texttt{cmd/gt/testdata/bench/sample.\{go,py,js,tsx\}}). The v2 corpus
supersedes these for the purposes of §\ref{sec:eval:compression}.}

\paragraph{Corpus.} The broader corpus for OQ-AST-1 (token-budget
conformance), OQ-AST-2 (latency), and OQ-AST-5 (end-to-end) consists
of: (a) the Blackrim repository itself ($\sim$3\,175 source files
across Go, Python, JavaScript, and TypeScript, excluding worktrees,
testdata, and vendored directories); (b) a public-repo polyglot sample
pinned in \texttt{scripts/corpus.txt}: the Go standard library, the
CPython standard library subset, the React source tree, the TypeScript
compiler, and Kubernetes. Files with \texttt{outline:skip} or in
vendored directories are excluded. This broader corpus is reserved for
the pending OQs; it was not used for the compression-ratio measurement
in \cref{sec:eval:compression}.

\paragraph{Telemetry.}{\sloppy
Adoption-trajectory data (OQ-AST-3) is sourced
from the \emph{federated} outline-events sink at
\texttt{\$BLACKRIM\_HOME/.beads/telemetry/outline-events.jsonl}
(default \texttt{\$HOME/Code/blackrim/}), to which the PreToolUse
hook (\cref{sec:system:discipline}) writes a copy of every qualifying
event across the Blackrim factory \emph{and} every factory project the
user works in (those onboarded via \texttt{blackrim-project on}). The
hook also keeps a per-project copy at
\texttt{<project\_root>/.beads/telemetry/outline-events.jsonl}
for project-local dashboards, but the central sink is the
paper's load-bearing source. Schema fields:
\texttt{ts, tool, file, loc, outline\_called\_prior, verdict, agent,
transcript\_id, project\_root}. The
\texttt{project\_root} tag lets us group adoption by factory vs.\
consumer projects and detect whether the discipline ports cleanly
across the \texttt{blackrim-project on} boundary (relevant to
\cref{sec:discussion:limits}). End-to-end traces (OQ-AST-5) are
sourced from \texttt{.beads/telemetry/invocations.jsonl} in the
corresponding project root, joined to outline events by transcript ID.\par}

\subsection{Compression ratios by language}
\label{sec:eval:compression}

\Cref{fig:compression} reports the mean token-savings ratio
$1 - \ratio(\file)$ per language across the 10-file per-language corpus
(v2). All four languages clear the per-language floors enforced by the
in-tree benchmark
assertion\footnote{\texttt{cmd/gt/compress\_structure\_bench\_test.go}}:
Go~65.2\% (floor 50\%), Python~72.3\% (floor 20\%),
JavaScript~78.1\% (floor 50\%), TypeScript~75.7\% (floor 50\%).

\begin{figure}[ht]
\centering
\input{figures/compression-by-lang}
\caption{Mean token-savings ratio $1 - \ratio(\file)$ by language on
the 40-file polyglot corpus (v2; $n{=}10$ per language). Per-language
floors enforced by the benchmark suite are: Go 50\%, Python 20\%,
JavaScript 50\%, TypeScript 50\% \cite{blackrim_adr_0002}; measured
values clear the floors in all four languages. Go's savings (65.2\%)
are lower than in the prior $n{=}1$ fixture (82.0\%) because the v2
corpus includes large files (e.g.\ \texttt{eval/runner.go} at
1098~LoC) whose public-function signatures occupy more outline space
relative to body.}
\label{fig:compression}
\end{figure}

\paragraph{Why the per-language variance.}
Python's highest savings (72.3\%) reflect the language's indented
block structure: function and class bodies are large relative to their
headers, so dropping body lines yields a small outline. JavaScript
achieves the highest savings in v2 (78.1\%), driven by the React
source corpus which has many small single-component files where the
export declaration and function header are the bulk of the structural
surface. Go's savings are lower in v2 (65.2\%) than in the prior
$n{=}1$ measurement (82.0\%); the v2 corpus intentionally includes
large files with many exported functions whose full signatures are
preserved by the outline, reducing the body-drop ratio. TypeScript
(75.7\%) is stable across v1 and v2 because the v2 files are drawn
from the TypeScript compiler's services layer, which has a similar
ratio of declaration headers to body as the original fixture.
The variance is design-intended; the floor assertion is the contract
and all four languages pass it.

\paragraph{Body-leak canaries.} A separate test in the same harness
asserts that no function body bytes appear in the outline output: a
known sentinel string (\texttt{BENCH\_BODY\_SENTINEL\_DO\_NOT\_LEAK})
is placed in the test fixtures and the bench fails if it appears in
the compressed output. This is a strong cross-cutting guard that a
regression in the per-language extractor cannot silently bypass.

\subsection{Token-budget conformance}
\label{sec:eval:budget}

\Cref{fig:budget} reports measured conformance to the token budget
$B = 300$ on Phase 1a (Go-only corpus). Across all three repositories,
\textbf{59.3\%} of files stay within budget (600 files sampled,
200 per repo). Per-repository breakdown: Blackrim (jsgerman-oss/blackrim.dev@HEAD)
achieves \textbf{76.5\%} with a $p_{50}$ of 239~tokens; Go standard
library (golang/go@release-branch.go1.22) achieves \textbf{67.5\%}
with $p_{50} = 247$~tokens; Kubernetes
(kubernetes/kubernetes@release-1.30) achieves only \textbf{34.0\%}
with $p_{50} = 413$~tokens.

The Kubernetes result marks an empirical divergence from the design
assumption. Hand-written Go code (Blackrim, stdlib) meets or approaches
the 95\% target; codegen-heavy API types (Kubernetes \texttt{pkg/apis})
drive the outline past budget routinely, even after truncation of
nested private constructs. This is not a failure of the algorithm---the
truncation cascade (\cref{sec:algorithm:truncation}) is functioning
as designed---but a signal that the budget contract needs per-repo tuning
or the truncation heuristic needs refinement for codegen-dominated files.

We propose three remedies: (a) a per-repo budget override in
\texttt{ship.toml [outline] budget\_tokens} to allow operators to set
$B$ per corpus; (b) retention of the \texttt{outline-budget-exceeded}
log event (already instrumented in \cref{sec:system}) for inspection
of which file classes exceed budget most; (c) tightening the truncation
cascade to truncate public-declaration signatures (currently preserved)
when nested constructs fail to fit, rather than halting at the private-construct
boundary. Caveat: the outline token estimate uses the $\text{chars}/4$
approximation (\cref{sec:design:tokenizer}); actual counts via the
\texttt{cl100k} encoder vary by $\pm 5{-}15\%$ and are captured in the
raw telemetry.

\subsection{Outline emission latency}
\label{sec:eval:latency}

\Cref{fig:latency} reports measured warm-run latency across file sizes
on Phase 1a (Blackrim Go files only). The 500\,ms target is comfortably
met: median latency (warm $p_{50}$) ranges from \textbf{37.81\,ms}
on files of 100--200~LoC to \textbf{41.03\,ms} on files of 1000--2000~LoC,
an order of magnitude below target. The 95th-percentile warm $p_{95}$
ranges from 40.99\,ms to 42.67\,ms, still subsecond. This performance
profile reflects the dominant cost: tree-sitter AST parse and traversal.
No sample files in Phase 1a exceeded 2000~LoC; corpus extension to
the stdlib and Kubernetes corpora, and bucketing up to 5000~LoC, is
deferred to blackrim-d8ya (Phase 1b, Python/JS/TS).

The first-token latency (the user-perceived cost in the prepend-to-Read
path; \cref{sec:system:hook}) is not separately reported in this
measurement; it is one emission call among the warm runs and is
thus included in the $p_{50}$ and $p_{95}$ statistics. The overhead
of prepending a brief Markdown outline to a \texttt{Read} result is
amortized across the tool call and is negligible ($< 50$\,ms,
already within the $p_{95}$ bounds above).

\subsection{Outline-discipline adoption trajectory}
\label{sec:eval:adoption}

\textbf{Pending --- OQ-AST-3.} The pre-registered hypothesis is that
the warn-mode-with-auto-prepend hook will drive adoption to $\geq
80\%$ within 14 days of rollout across all transcripts, sufficient to
trigger the promote-to-block trigger
(\cref{sec:system:discipline}). Empirically open:
\tothink{rolling 7-day hit-rate over the rollout period, plotted with
the 80\% promotion threshold marked; alongside, a histogram of
per-transcript hit-rate showing whether adoption is uniform across
agents or concentrated in a few well-disciplined ones.}

\subsection{Compile-gate false-negative rate}
\label{sec:eval:gate}

\textbf{Pending --- OQ-AST-4.} The pre-registered hypothesis is that
$\gate$ satisfies \cref{def:false_negative} on at least 99\% of
candidate diffs across a curated rename / extract / fix-imports
corpus, with $\leq 5\%$ false-negative rate (gate rejects diffs that
ground-truth review would have accepted). Empirically open:
\tothink{confusion matrix over a 200-diff curated set, broken out by
intent and language.} The 5\% false-negative target is the
conservatism budget; below it, the gate is correctly biased; above
it, the gate is over-conservative and operators will route around
it.

\subsection{End-to-end token reduction on real agent traces}
\label{sec:eval:e2e}

\textbf{Pending --- OQ-AST-5.} This is the central pending claim
\emph{for the paper as a whole}: the per-file compression ratio is
measured (\cref{sec:eval:compression}), but the integrated token cost
across a real agent task --- one that touches many files, may need
to read some bodies after their outlines, and pays the hook
overhead --- is what determines whether the design wins in
deployment.

Pre-registered comparison: paired traces of the same Explore-tier
tasks under (a) baseline (no hook, agents read files in full); (b)
outline-discipline enabled; same agent, same task, same prompt, same
model tier. Outcome metrics: total input tokens per task, wall-clock
to first answer, downstream-task-success rate
(\cref{sec:problem:success}).

Empirically open: \tothink{paired-trace dataset across $N_\text{task}
\geq 50$ tasks, with per-task token deltas, latency deltas, and
success deltas reported with bootstrap 95\% CIs.}

\subsection{Reproducibility}
\label{sec:repro}

Every quantitative claim in this section is backed by a CSV in
\texttt{data/} produced by a script in \texttt{scripts/}. The full
reproduction pipeline is:

\begin{lstlisting}[language=bash]
# Reproduces 6.2 (currently measured; uses local corpus-files/ corpus)
python scripts/pull-compression-ratios.py \
    > data/raw/compression-ratios.jsonl
python scripts/aggregate-by-language.py \
    < data/raw/compression-ratios.jsonl \
    > data/aggregated/by-language.csv
# Legacy n=1 single-fixture mode (pre-v2):
#   python scripts/pull-compression-ratios.py --legacy \
#       --blackrim-root=$BLACKRIM_ROOT > ...

# Reproduces 6.4 (pending; harness ready)
python scripts/measure-outline-latency.py \
    --blackrim-root=$BLACKRIM_ROOT \
    > data/aggregated/outline-latency.csv

# Reproduces 6.5 (pending; needs accumulated telemetry)
python scripts/pull-outline-telemetry.py \
    --telemetry=$BLACKRIM_ROOT/.beads/telemetry/outline-events.jsonl \
    > data/aggregated/outline-events.csv

make
\end{lstlisting}

The pipeline is deterministic given (a) a Blackrim checkout at a
fixed commit and (b) the test corpus pinned in
\texttt{scripts/corpus.txt}. We anchor the published version to a
tagged Blackrim commit; the companion repository
\cite{blackrim_paper_repo} hosts the artefacts.
