## Tasks
- [ ] **§8 Discussion.** Rewrite `sections/08-discussion.tex`.
  Structure as 5 subsections:
  1. *Why the two primitives compose.* Argue from §3's cost model
     that prefix caching, plan caching, and per-turn routing
     attack three different cost dimensions: prefix caching
     reduces the per-call multiplier on cache-warm calls, plan
     caching skips calls entirely, routing reduces the per-call
     dollar rate. Each is fidelity-preserving along a different
     axis. Composition = multiplicative, not additive.
  2. *What we did NOT do.* This paper deliberately scoped out:
     (a) DistilBERT / heavier router architectures, (b) cascade
     routing with semantic entropy, (c) online (vs offline-
     replay) plan-cache deployment, (d) cross-system
     generalisation. Frame each as a deliberate scope decision,
     not an oversight.
  3. *Single-system case study limitation.* The measurements
     come from one production multi-agent system (Blackrim). The
     architecture details (Admiral + 6 crew + workers, opus-on-
     main, 5-min cache TTL) are common but not universal. Argue
     why the methodology generalises (the cost model is system-
     agnostic; only the constants change).
  4. *Threats to validity.* (a) Test set size for routing (10
     turns); (b) replay-only evaluation for plan caching (no
     online drift measurement); (c) cost-model assumes constant
     prefix-cache hit rate within a window; (d) hand-labelled
     exemplars subject to author bias.
  5. *When NOT to apply these primitives.* If main-thread cost
     share is <50%, routing's headroom is limited. If dispatch
     plans are highly heterogeneous (low signature reuse), plan
     caching adds latency without value. Help practitioners
     decide.
