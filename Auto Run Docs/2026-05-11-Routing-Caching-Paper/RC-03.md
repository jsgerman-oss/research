# RC-03 — Routing test set + harness (design, no live API)

## Goal
Ship the **routing eval harness** that §7 "Routing evaluation" currently
leaves as a `\tothink`. Hand-label 50 real Blackrim turns with their
should-be-tier target (haiku / sonnet / opus), build a CLI to score any
candidate router against the labels, and run it end-to-end in dry-run.
The actual router implementation + live measurement is RC-04.

## Tasks
- [ ] Sample 50 main-thread turns from Blackrim session history.
  Approach: take the most recent N session files, parse turn-level
  records, filter to main-thread user→assistant exchanges, sample
  uniformly. Save raw turn text + metadata (timestamp, session-id,
  turn-index) to `scripts/eval-routing/turns/turn-NN.json` (one per
  turn). Strip any sensitive content (paths, secrets) but keep the
  semantic meat.
- [ ] Hand-label each turn with `should_be_tier`:
  - **haiku** if the turn is read-mostly / lookup / simple formatting
    (e.g. "list files matching X", "echo this back").
  - **sonnet** if reasoning is required but not deep architecture
    decisions (e.g. small refactor, test write, doc update).
  - **opus** if the turn requires deep architecture reasoning, multi-
    file refactor planning, or high-stakes correctness (e.g.
    migrations, security review, API design).
  Use `model-cost-quality-landscape.md` Section "tier boundaries"
  as the rubric. Store labels in `scripts/eval-routing/labels.yml`
  as `[{turn_id, should_be_tier, rationale}]`. Aim for roughly
  20% haiku / 50% sonnet / 30% opus (don't force-balance if natural
  distribution differs — note the imbalance).
- [ ] Build `scripts/eval-routing/run.py` — the CLI. Accepts:
  `--router <name>` (one of `always-opus`, `always-sonnet`,
  `random-uniform`, `semantic-similarity` — the latter is the
  proposed router, stub returning haiku for now), `--turns-dir
  scripts/eval-routing/turns/`, `--labels
  scripts/eval-routing/labels.yml`, `--out
  data/aggregated/routing-eval-<router>.csv`. For each turn:
  call `route(turn_text) -> predicted_tier`, emit CSV row with
  `turn_id, gold_tier, pred_tier, correct, cost_saved_vs_opus_usd,
  cost_of_mistake_usd` (use pricing table from RC-02). Add a final
  summary CSV `data/aggregated/routing-summary-<router>.csv` with
  per-tier {precision, recall, F1} + overall accuracy + total
  cost-saved + total cost-of-mistakes.
- [ ] Run two baseline routers end-to-end:
  - `python scripts/eval-routing/run.py --router always-opus
     --turns-dir scripts/eval-routing/turns/
     --labels scripts/eval-routing/labels.yml
     --out data/aggregated/routing-eval-always-opus.csv`
  - `python scripts/eval-routing/run.py --router random-uniform ...`
  These run without API calls — they're pure baselines. The
  always-opus baseline should have 100% recall on opus-gold turns
  and 0% on others. Random should be ~33% accuracy. If either
  baseline is wildly off, the harness has a bug.
- [ ] Update §7 routing-evaluation paragraph with a placeholder
  reference to the baseline numbers ("Baselines: random-uniform
  achieves X% accuracy, always-opus 100% recall on opus-gold,
  Z% overall waste."). The proposed-router number comes in RC-04.
  Drop the `\tothink` only after RC-04.
- [ ] Document the harness in `scripts/eval-routing/README.md`:
  how labels were assigned (cite `model-cost-quality-landscape.md`
  rubric), how to add new routers (subclass `Router`), how to
  interpret the summary CSV.
- [ ] Commit: `eval(rc): routing eval harness + 50-turn hand-
  labelled test set + baseline (always-opus, random) results`.

## Acceptance criteria
- `scripts/eval-routing/turns/turn-{01..50}.json` exist with
  real turn content (sensitive content scrubbed).
- `scripts/eval-routing/labels.yml` has 50 entries.
- `data/aggregated/routing-eval-always-opus.csv` and
  `routing-eval-random-uniform.csv` exist with 50 data rows each.
- Per-router summary CSVs exist with sensible per-tier metrics.
- §7 references the baselines (placeholder for the proposed
  router pending RC-04).

## Style / care
- Labels are the foundation of the entire eval — be conservative.
  When in doubt between sonnet and opus, label opus; the proposed
  router being "wrongly cautious" (routed to opus when sonnet
  would have sufficed) is a smaller cost than the inverse. Note
  this conservatism in the README.
- Don't peek at the proposed router's output before labelling —
  contamination risk.

## Out of scope

- Confidence-based escalation (a future-work direction; will
  appear in §9 not in this harness).
